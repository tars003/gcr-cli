# GCR CLI - Design Document v2

**Last Updated:** 2026-01-14
**Status:** Design Phase

## Overview

GCR (Git ClickUp Release) is a TUI-based CLI tool that automates PR creation, code review, and release management by integrating GitHub, ClickUp, and AI code review agents.

## Team & Roles

- **Ajay** - Tech lead & developer
- **Darshan** - Developer
- **Kushal** - Product manager
- **Swapnil** - QA
- **Sagar** - Client

## Problem Statement

Current workflow is manual and disconnected:
- Manual compilation of ClickUp tickets for weekly client updates
- Separate tools for PR review (Claude Code + gh CLI)
- No automated linking between GitHub PRs and ClickUp tasks
- Time-consuming release note compilation from multiple PRs
- Clunky communication chain for urgent issues

## Solution

A context-aware TUI application that:
1. **Developers** create PRs with automatic AI review and ClickUp task linking
2. **Ajay** reviews/merges PRs with inline code review and generates partial release notes
3. **Ajay** compiles releases from staging → production with aggregated release notes

### Integration Adapters

**Agent Adapter**
- Generic interface for AI providers (Claude, OpenAI, etc.)
- Loads prompts from `src/prompts/` directory
- Prompt-based interface: CLI owns prompts, agent executes

**GitHub Adapter**
- Wraps `gh` CLI and GitHub API
- PR operations, comment posting, release creation
- Query merged PRs for release compilation

**ClickUp Adapter**
- REST API client
- Fetch tasks from configured lists
- Optional status updates (configurable)

**Git Adapter**
- Shell command wrapper
- Branch detection, diff generation
- Remote URL parsing

### State Management

**Per-project config:** `.gcr/config.json` (version controlled)
```json
{
  "projectName": "MyProject",
  "branches": {
    "production": "production",
    "staging": "staging"
  },
  "clickup": {
    "lists": [
      {"name": "Backlog", "id": "12345"},
      {"name": "Bugs", "id": "67890"}
    ],
    "autoUpdateStatus": false
  },
  "agent": {
    "provider": "claude",
    "reviewProfile": "relaxed",
    "customRules": [
      "console.logs are acceptable",
      "commenting out code is fine",
      "tests not required"
    ]
  },
  "versioning": "semver"
}
```

**Global credentials:** `~/.config/gcr/credentials.json` (not version controlled)
```json
{
  "clickup": {
    "apiKey": "..."
  },
  "agents": {
    "claude": {"apiKey": "..."},
    "openai": {"apiKey": "..."}
  }
}
```

**No local state files** - GitHub is the source of truth for PR/release tracking

## Workflows

### 1. PR Creation Flow (Developer)

**Trigger:** Developer runs `gcr` on feature/fix/hotfix branch

**Steps:**
1. **Branch Detection & Target Selection**
   - Detect branch name pattern
   - If `hotfix/*`: Prompt "Merge to staging or production?"
   - Otherwise: Target staging

2. **ClickUp List Selection**
   - TUI menu: Select from configured lists
   - Fetch all tasks from selected list

3. **AI Code Review**
   - Generate diff against target branch
   - Agent reviews with custom rules from config
   - Store review results for later

4. **Task Matching**
   - Agent analyzes diff + ClickUp tasks
   - Suggests relevant tasks (bias toward recall over precision)
   - TUI multi-select: Developer confirms tasks

5. **PR Creation**
   - Create PR via `gh` CLI
   - Post Comment #1: Linked ClickUp tasks with URLs
   - Post Comment #2: AI review findings (if any)

6. **ClickUp Status Update (Optional)**
   - If `autoUpdateStatus: true`: Update tasks "IN PROGRESS" → "IN QA"

**Error Handling:** Fail-fast with rollback. If any step fails, delete draft PR and show error.

### 2. PR Review Flow (Ajay)

**Trigger:** Ajay runs `gcr` on staging/production branch

**Steps:**
1. **PR Selection**
   - Query open PRs targeting current branch
   - TUI list: Select PR to review

2. **Sequential Review Display**
   - Section 1: PR metadata (title, author, branch)
   - Section 2: Commits list (scrollable)
   - Section 3: AI review findings (if any)
   - Section 4: Developer responses (if any)
   - Navigate with arrow keys between sections

3. **Merge Decision**
   - Action menu: Merge / Request Changes / Cancel

4. **Partial Release Notes Generation (If Merging)**
   - Agent generates bullet points from diff + ClickUp tasks
   - TUI editor: Ajay reviews/edits notes
   - Post Comment #3: Partial release notes
   - Merge PR

### 3. Release Flow (Ajay)

**Trigger:** Ajay runs `gcr` on staging branch (Wed/Sat)

**Steps:**
1. **PR Selection for Release**
   - Prompt: "Create release from last release? Yes / No"
   - If Yes: Query PRs merged into staging since last GitHub release
   - If No: Ask for number of days → query last X days
   - TUI checklist: Multi-select PRs to include

2. **Release Notes Compilation**
   - Extract partial release notes from selected PR comments
   - Agent compiles into cohesive release notes
   - TUI editor: Ajay reviews/edits final notes

3. **Version Selection**
   - Analyze PR types (breaking/feature/fix)
   - Agent suggests semantic version bump
   - TUI: Ajay confirms or overrides version

4. **Release Creation**
   - Merge staging → production
   - Create GitHub release with compiled notes + ClickUp task links
   - Tag with version number

## AI Agent Interface

### Prompt Template System

CLI owns all prompts in `src/prompts/` directory. Agent interface is generic completion:

```typescript
interface Agent {
  complete(prompt: string, context: Record<string, any>): Promise<string>;
}
```

**Prompts:**
- `review-code.txt` - Code review with custom rules
- `match-tasks.txt` - Match diff to ClickUp tasks
- `generate-release-notes.txt` - Generate release notes from PRs
- `suggest-version.txt` - Suggest semantic version bump

**Benefits:**
- Centralized prompt management
- Easy to modify/version prompts
- Provider-agnostic

### Review Profile System

**Preset Profiles:**
- `strict` - Comprehensive review, all best practices
- `balanced` - Standard review, reasonable checks
- `relaxed` - Light review, focus on critical issues only

**Custom Rules:**
Array of natural language rules that agent interprets:
```json
"customRules": [
  "console.logs are acceptable",
  "commenting out code is fine",
  "tests not required"
]
```

## Git Branching Strategy

### Hybrid Model

**Normal Feature Development:**
- `feature/*` or `fix/*` branches → staging
- Staging accumulates changes during week
- Wed/Sat: staging → production (scheduled release)

**Urgent Hotfixes:**
- `hotfix/*` branches → staging OR production (developer chooses)
- Can bypass staging for critical fixes
- Still tracked in release notes

**Branch Detection:**
CLI detects branch naming pattern and prompts appropriately.

## ClickUp Integration

### Task Status Flow

**Statuses:** "TO DO" → "IN PROGRESS" → "IN QA"

**Automation (if enabled):**
- PR Created: "IN PROGRESS" → "IN QA"
- After "IN QA": Swapnil (QA) takes over manually

**Configuration:**
```json
"clickup": {
  "autoUpdateStatus": false  // Opt-in
}
```

### Task Linking

**In PR Comment #1:**
```markdown
## Linked ClickUp Tasks

- [CU-abc123] Implement video ad music library
  https://app.clickup.com/t/abc123
- [CU-def456] Fix audio playback issue
  https://app.clickup.com/t/def456
```

**In Release Notes:**
Aggregated from all PRs in release with task links preserved.

## Error Handling

### Fail-Fast with Rollback

**Philosophy:** Each workflow is a transaction - either fully succeeds or fully rolls back.

**Implementation:**
1. Track created resources in memory during workflow
2. On error at any step:
   - Delete draft PR if created
   - Don't post comments
   - Don't update ClickUp status
3. Show error message with specific failure reason
4. Exit cleanly

**User Experience:**
```
┌─────────────────────────────────────┐
│  Error: Failed to create PR         │
├─────────────────────────────────────┤
│  ClickUp API request timed out      │
│                                     │
│  No changes were made.              │
│                                     │
│  [Enter] Retry  [Q] Quit            │
└─────────────────────────────────────┘
```

**Critical:** No partial state left behind.

## Technology Stack

**Language:** Node.js / TypeScript

**Key Libraries:**
- `ink` - React for CLIs (TUI framework)
- `ink-select-input` - Menu/list selection
- `ink-text-input` - Text input
- `ink-spinner` - Loading indicators
- `commander` - CLI argument parsing (for `gcr init`)
- `axios` - HTTP client for APIs
- `simple-git` - Git operations

**External Dependencies:**
- `gh` CLI - Must be installed and authenticated
- Git - For repository operations

**Distribution:**
- NPM package (`npm install -g gcr`)
- Single entry point: `gcr` command

## Versioning Strategy

**Semantic Versioning:** `major.minor.patch`

**Version Bumping Logic:**
- Breaking changes (hotfix to production, major refactor): Major bump
- New features: Minor bump
- Bug fixes only: Patch bump

**AI Assistance:**
- Agent analyzes PR types in release
- Suggests version bump
- Ajay confirms or overrides in TUI

**Version Storage:**
- GitHub releases and git tags only
- No version file in repo

## Data Flow

### PR Creation Flow

```
Developer (feature branch)
  ↓
gcr detects branch → shows "Create PR" option
  ↓
Select ClickUp list → Fetch tasks
  ↓
Git diff → Agent reviews code
  ↓
Agent matches tasks to diff → Developer confirms
  ↓
gh CLI creates PR
  ↓
Post comments: (1) Tasks (2) Review
  ↓
Optional: Update ClickUp task status
```

### PR Review Flow

```
Ajay (staging branch)
  ↓
gcr shows "Review PR" option → Select PR
  ↓
Fetch PR data: commits, comments, reviews
  ↓
Display sequentially: metadata → commits → AI review → responses
  ↓
Ajay decides: Merge / Request Changes / Cancel
  ↓
If merge: Agent generates partial release notes
  ↓
Ajay edits notes → Post as comment → Merge PR
```

### Release Flow

```
Ajay (staging branch)
  ↓
gcr shows "Create Release" option
  ↓
"Since last release?" → Yes: query since last tag / No: ask days
  ↓
Show PRs merged to staging → Ajay multi-selects
  ↓
Extract partial notes from PR comments
  ↓
Agent compiles release notes → Ajay edits
  ↓
Agent suggests version → Ajay confirms/overrides
  ↓
Merge staging → production
  ↓
Create GitHub release with notes + version tag
```
